# 3D Assembly Viewer: Research & Implementation Plan

Interactive 3D viewer for the Requencer's three-board sandwich stack (faceplate + control + main), accessible from the web preview debug menu. Supports orbit rotation and an exploded-view slider to pull boards apart for assembly inspection.

## 1. Goal

Render the three PCBs with mounted components in 3D, directly in the browser. Key interactions:
- **Orbit/rotate** the assembly freely
- **Explode/collapse** slider to separate boards along Z-axis (like hardware assembly animations)
- **Identify assembly issues** — component clearance, connector alignment, board-to-board fit

## 2. Available Data

### Board STEP Files (full 3D geometry with components)
Generated by `hardware/boards/scripts/export_3d_assembly.py`:
- `hardware/boards/build/3d/faceplate.step`
- `hardware/boards/build/3d/control.step`
- `hardware/boards/build/3d/main.step`

### Stack-Up Z Positions (from `export_3d_assembly.py`)
| Board | Z Position | Notes |
|-------|-----------|-------|
| Faceplate | 0 mm | Front face, 1.6mm thick |
| Control board | −3.2 mm | 1.6mm air gap behind faceplate |
| Main board | −13.3 mm | 8.5mm connector height + 1.6mm control PCB |

PCB origin offset from faceplate: (2.0 mm, 9.5 mm).

### Component 2D Positions (`web/src/panel-layout.json`)
- All user-facing components with x_mm, y_mm coordinates
- Footprint bounding boxes (body + courtyard) for jacks, switches, encoders
- LCD cutout geometry (82.5 x 52.0 mm opening, 85.5 x 54.94 mm glass)
- Board dimensions: faceplate 181.88 x 127.5 mm, PCB 177.88 x 107.5 mm

### Component Metadata (`hardware/boards/component-map.json`)
- Atopile address → component type/group/label mapping
- Display specs (JC3248A035N-1, 480x320, ST7796S driver)
- Physical dimensions for all component types

### Generated 3D Models (`hardware/boards/scripts/generate_3d_models.py`)
Custom parametric STEP files for components without manufacturer models:
- **TC002-RGB switch**: 7x7x3.5 mm body + 3.2 mm actuator, 12 mm above PCB
- **PGA2350 module**: 25.4x25.4x1.6 mm with RP2350B chip

### 3D Model Assignments (`hardware/boards/scripts/add_3d_models.py`)
Maps KiCad footprints to STEP files with rotation/offset overrides. Sources:
- KiCad 9 standard library (`${KICAD9_3DMODEL_DIR}/...`)
- Local parts (`hardware/boards/parts/*/`)

## 3. Implementation Approaches

### Option A: Three.js with STEP → glTF Conversion (Recommended)

**Pipeline:** STEP → glTF (offline) → Three.js (runtime)

1. **Offline conversion** — Use a Python script (e.g. `cadquery` or `OCP/opencascade` via `build123d`) to convert the three board STEP files into `.glb` (binary glTF) files. These go into `web/public/models/`.
2. **Runtime** — Three.js loads the `.glb` files and renders them with orbit controls + an explode slider.

**Pros:**
- glTF is the "JPEG of 3D" — optimized for web, fast loading
- Three.js is mature, well-documented, ~150 KB gzipped
- Offline conversion keeps the runtime simple
- No STEP parsing in the browser (STEP files can be huge)

**Cons:**
- Requires a conversion step (can be added to `make hw-all` pipeline)
- Three.js is a new dependency

**Conversion approach:**
```python
# Using opencascade via cadquery or OCP
# STEP → tessellated mesh → glTF
import cadquery as cq
from cadquery import exporters

result = cq.importers.importStep("control.step")
exporters.export(result, "control.glb")
```

Or use the `ifcopenshell` / `pythonocc-core` / `trimesh` ecosystem. Another option is the CLI tool `assimp` which can convert STEP → glTF.

### Option B: Three.js with Runtime STEP Loading (occt-import-js)

**Pipeline:** STEP files served directly → parsed in browser via `occt-import-js` (OpenCascade compiled to WASM)

**Pros:**
- No offline conversion step
- Always up-to-date with latest STEP files

**Cons:**
- `occt-import-js` WASM is ~8 MB — heavy initial load
- STEP parsing in the browser is slow (seconds per file)
- Less mature than the glTF path

### Option C: Simplified Parametric View (No STEP files)

**Pipeline:** Use `panel-layout.json` data to render boxes/cylinders for components

1. Render board outlines as flat colored rectangles at correct Z positions
2. Place simple 3D primitives (cylinders for jacks/encoders, boxes for switches/ICs)
3. Use dimensions from `panel-layout.json` constants and `component-map.json`

**Pros:**
- Zero external 3D model dependencies
- Very fast loading, small bundle size
- Easy to implement and maintain
- Data already available in JSON

**Cons:**
- Low visual fidelity — won't show actual component shapes
- Less useful for identifying real clearance issues
- Doesn't leverage the existing STEP models we already generate

### Option D: Hybrid — Parametric + Optional STEP Loading

Start with Option C (parametric boxes) as the default, with an optional "load detailed models" button that fetches pre-converted glTF files if available.

**Pros:**
- Instant initial load with useful geometry
- Progressive enhancement with real models
- Graceful fallback if models aren't built yet

## 4. Recommended Approach: Option A (STEP → glTF + Three.js)

We already generate the STEP files via `export_3d_assembly.py`. Adding a glTF conversion step is straightforward and gives the highest visual fidelity with reasonable bundle size.

### New Files

```
web/src/ui/assembly-viewer.ts    — Three.js scene setup, model loading, explode control
web/public/models/                — glTF files (gitignored, generated)
hardware/boards/scripts/export_gltf.py  — STEP → glTF converter
```

### Debug Menu Integration

Add a "3D View" button to the existing debug menu (`web/src/ui/debug-menu.ts`). Clicking it:
1. Overlays a full-screen `<canvas>` (or a modal with a canvas)
2. Initializes the Three.js scene
3. Loads the three `.glb` files
4. Shows orbit controls + explode slider + close button

### Three.js Scene Setup

```typescript
import * as THREE from 'three';
import { OrbitControls } from 'three/addons/controls/OrbitControls.js';
import { GLTFLoader } from 'three/addons/loaders/GLTFLoader.js';

// Scene with ambient + directional lighting
// Camera positioned to see full assembly
// OrbitControls for rotation/zoom/pan

// Load 3 boards as separate groups
const boards = ['faceplate', 'control', 'main'];
const zPositions = [0, -3.2, -13.3]; // mm

// Explode slider: multiplies Z offsets by a factor (1x = assembled, 3-5x = exploded)
function setExplodeFactor(factor: number) {
  boards.forEach((board, i) => {
    board.group.position.z = zPositions[i] * factor;
  });
}
```

### Explode/Collapse Interaction

- **Slider** (range input): 1.0 (collapsed/assembled) → 5.0 (fully exploded)
- Multiplies the Z spacing between boards
- Smooth CSS transition or TWEEN animation when snapping to presets
- Optional: click individual board labels to isolate/highlight one board

### Build Pipeline Addition

Add to `Makefile`:
```makefile
hw-export-gltf:  ## Convert STEP → glTF for web 3D viewer
    python hardware/boards/scripts/export_gltf.py
```

And chain it: `hw-all` → includes `hw-export-gltf` after `export_3d_assembly.py` runs.

### Dependencies

```json
{
  "three": "^0.170.0"
}
```

Three.js is ~600 KB uncompressed, ~150 KB gzipped. Tree-shaking with Vite will help. Since this feature is behind a debug menu button, the glTF loader and Three.js can be **dynamically imported** to avoid impacting initial page load:

```typescript
const loadViewer = async () => {
  const { createAssemblyViewer } = await import('./assembly-viewer');
  createAssemblyViewer(container);
};
```

## 5. STEP → glTF Conversion Options

| Tool | Method | Notes |
|------|--------|-------|
| `cadquery` + `cq-export` | Python | Popular, good STEP support |
| `pythonocc-core` | Python (OpenCascade bindings) | Full OCCT power, heavier install |
| `build123d` | Python | Modern cadquery successor |
| `FreeCAD` CLI | `freecadcmd` batch | Full CAD kernel, scriptable |
| `OpenCascade` → `trimesh` | Python | Tessellate with OCC, export with trimesh |
| Docker image | Pre-built with OCCT + Python | Avoids local install complexity |

Since we already have a Docker-based hardware build pipeline (`hardware/docker/`), the cleanest approach is to add the glTF conversion to the existing Docker image and run it as part of `make hw-all`.

## 7. Open Questions

1. **STEP file size** — How large are the generated STEP files? If the glTF files are >10 MB each, we may want mesh decimation or LOD.
2. **Color/material data** — Do the STEP files preserve per-component colors (copper, FR4 green, component black)? If not, we'd need to apply materials in Three.js based on component type.
3. **Dynamic import vs. separate page** — Should the 3D viewer be a modal overlay in the main app, or a separate route/page? Modal overlay keeps it integrated with the debug menu.
4. **Faceplate transparency** — It would be useful to render the faceplate semi-transparent so you can see the control board components behind it. This is easy in Three.js (material.opacity).
5. **Conversion tool availability** — Need to verify which Python STEP→glTF tool works best in our Docker environment.
